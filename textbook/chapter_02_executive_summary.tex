\section{Problem Statement}
\label{sec:problem-statement}

\subsection{The Operational Reality}

The Faculty of Engineering at Beni-Suef University (BSU) accommodates approximately \textbf{2,000+ students} distributed across five academic departments: Preparatory (الفرقة الإعدادية), Electrical Engineering (with specializations in Electronics \& Communications and Power \& Machines), Mechanical Engineering, Civil Engineering, and Architectural Engineering. Each department operates across up to four academic levels, two semesters per year, and manages dozens of courses per term.

The administrative machinery supporting this academic structure relies on a patchwork of \textbf{manual processes, disconnected spreadsheets, and informal communication channels}. This is not merely an inconvenience---it is a systemic risk to data integrity, operational efficiency, and academic governance.

\subsection{Specific Pain Points}

The following localized problems were identified through direct observation and stakeholder consultation:

\begin{itemize}
    \item \textbf{Student Data Management (The Excel Chaos):} Student enrollment and academic records are maintained in standalone Excel files. When a department needs student lists, a manual extraction process begins. This introduces data duplication, version conflicts, and eliminates any single source of truth.
    \item \textbf{Grade Entry \& Approval (The Paper Trail):} Faculty members record grades on paper or personal spreadsheets, physically submitting them to Student Affairs for manual re-entry. This multi-hop flow creates transcription errors, lacks an audit trail, and severely delays grade visibility for students.
    \item \textbf{Attendance Tracking (The Paper Register):} Attendance is recorded on printed sheets passed around lecture halls. Records risk physical loss, offer no real-time visibility for students, and provide no analytics capability for early academic intervention.
    \item \textbf{Material Distribution (The USB Drive Problem):} Lecture slides and notes are distributed via USB drives, personal emails, or social media groups. There is no centralized repository, no access control based on enrollment, and no version management.
    \item \textbf{Administrative Isolation (The Silo Effect):} Administrative units operate in isolation. There is no shared platform for tracking doctor-to-course assignments, cross-referencing enrollment with grades, or maintaining institutional accountability.
\end{itemize}

\subsection{The Core Problem Formally Stated}

\begin{quote}
\textbf{Problem Statement:} The Faculty of Engineering at Beni-Suef University lacks a unified, role-based digital platform for managing its core academic operations---including student enrollment, grade management, attendance tracking, course material distribution, and administrative coordination---resulting in data fragmentation, operational inefficiency, accountability gaps, and delayed information access for all stakeholders.
\end{quote}

\subsection{Why Not Off-the-Shelf Solutions?}

Commercial Learning Management Systems (e.g., Moodle) and Student Information Systems (e.g., Banner) were evaluated but found unsuitable. The decision to build a custom solution was driven by three converging factors:

\textbf{1. Egyptian Academic Workflow Specificity} \\
The Egyptian engineering grading structure does not map cleanly to Western software models. The three-component system (Coursework / Written / Practical-Oral) with variable weights requires a configurable grading template system.

\begin{lstlisting}[language=Python, caption=Custom Grading Template Seeds mapping to Faculty Rules]
# From seed_production.py - Configurable Templates
("نموذج المواد النظرية (40-60-0)", 40, 60, 0),    # Theoretical
("نموذج المواد العملية (30-50-20)", 30, 50, 20),   # Practical
("نموذج المعامل (20-40-40)", 20, 40, 40),          # Lab-heavy
\end{lstlisting}

Furthermore, the institution requires a strict authentication flow where a student's National ID serves as their initial credential, forcing an immediate password change upon first login.

\begin{lstlisting}[language=Python, caption=Institutional Authentication Requirement Enforced in Backend]
# From users/models.py - First-Login Enforcement
def save(self, *args, **kwargs):
    if is_new and not self.is_superuser:
        if not self.password or self.password == '!':
            self.set_password(self.national_id or self.username)
        if self.role in ['STUDENT', 'STUDENT_AFFAIRS', 'STAFF_AFFAIRS', 'DOCTOR']:
            self.first_login_required = True
\end{lstlisting}

\textbf{2. Arabic-First Interface Requirement} \\
The system must function as a native Arabic application with full Right-to-Left (RTL) layout. It leverages the MUI RTL configuration and the "Cairo" font family to ensure optimal legibility, rather than applying a brittle localization layer over an English-first design.

\textbf{3. Deployment Constraints} \\
The university requires a self-hosted, container-based deployment free from recurring cloud licensing costs or external dependencies (like paid SMS gateways), allowing the system to operate entirely behind institutional firewalls.

\subsection{The Engineering Motivation}

Beyond solving operational issues, this project demonstrates the application of modern software engineering principles:
\begin{itemize}
    \item \textbf{System Thinking:} Unifying diverse stakeholders into a coherent architecture.
    \item \textbf{Trade-off Analysis:} Deliberate architectural choices (e.g., Session Auth over JWT, Monolith over Microservices) driven by context.
    \item \textbf{DevOps \& Security:} Implementing CI/CD, Docker orchestration, and Trivy security scanning as integral lifecycle components.
\end{itemize}

\section{Proposed Solution}
\label{sec:proposed-solution}

\subsection{System Overview}

The \textbf{Release BSU Engineering Portal} is a full-stack, role-based academic management platform. It centralizes core academic operations through four functional pillars:
\begin{enumerate}
    \item \textbf{Student Lifecycle Management}: Replaces scattered Excel files with a relational database managing enrollment, grades, and attendance.
    \item \textbf{Faculty Course Management}: Digital course assignments, lecture uploads, batch attendance, and interactive quiz generation.
    \item \textbf{Administrative Operations}: Multi-step workflows for Excel batch uploads, grade approvals, and immutable audit logging.
    \item \textbf{Public Information Portal}: Structured news and contact routing.
\end{enumerate}

\subsection{High-Level Architecture}

The system utilizes a three-tier containerized architecture, separating presentation, business logic, and data persistence.

\subsection{Preview of Architectural Decisions}

\begin{itemize}
    \item \textbf{SPA + REST API:} Complete frontend-backend decoupling allows the React client to iterate independently of the Django API.
    \item \textbf{Nginx as the Single Entry Point:} The browser never communicates directly with Django. Nginx routes traffic, serves static assets, and injects security headers.
    \item \textbf{Database Isolation:} The MySQL container maps no ports to the host machine. It exists solely on an internal Docker network, reachable only by the backend worker.
\end{itemize}
